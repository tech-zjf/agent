# 客服 Copilot 实现版本文档

## 1. 文档信息
- 当前版本：`v0.3.0-langchain`
- 日期：`2026-03-13`
- 维护模块：`apps/agent-service`、`apps/agent-desktop`
- 目的：记录从模板到“数据库化 + LangChain 化 MVP”的实现路径，便于回看与面试讲解。

## 2. 版本演进

### v0.1.0-mvp（已完成）
1. 完成客服 Copilot 业务闭环。
2. 采用内存仓储（DataStore）快速验证业务。

### v0.2.0-db（已完成）
1. DataStore 从内存改为 PostgreSQL + TypeORM Repository。
2. 新增数据库实体、迁移历史、CLI 命令。
3. 环境配置从“脚本替换文件”改为 `.env + 校验`。
4. 为 pgvector 预留字段（migration 已创建 vector 列）。

### v0.3.0-langchain（本次完成）
1. 新增统一 AI 适配层，业务层不再直接依赖具体模型 SDK。
2. 接入 LangChain 官方 `ChatMinimax`，可选启用结构化 AI 决策。
3. Copilot 从“纯规则拼接”升级为“检索 + 记忆 + AI JSON 决策 + 守卫兜底”。
4. 环境变量补齐 AI 配置，支持显式开关与启动时强校验。

## 3. 本次实现目标
1. 不把 MiniMax 的参数和 SDK 直接耦合进业务层。
2. 保持当前接口和前端交互不变，平滑升级为 LangChain 驱动。
3. 让项目现在能用 MiniMax，后续也能继续接千问、豆包等其他模型。

## 4. 关键实现步骤

### Step 1：补齐 AI 配置体系
- 动作：在 `src/config/configuration.ts` 增加 `ai.chat` 与 `ai.providers.minimax`。
- 新增配置：
1. `AI_CHAT_ENABLED`
2. `AI_CHAT_PROVIDER`
3. `AI_CHAT_MODEL`
4. `AI_CHAT_TEMPERATURE`
5. `AI_CHAT_MAX_TOKENS`
6. `AI_CHAT_TIMEOUT_MS`
7. `MINIMAX_API_KEY`
8. `MINIMAX_GROUP_ID`
9. `MINIMAX_BASE_URL`
- 价值：
1. 启用/关闭 AI 明确可控。
2. 模型切换不需要改业务代码。

### Step 2：新增 AI 适配层
- 新增目录：`src/modules/ai`
- 新增内容：
1. `types/ai-provider.ts`：定义 provider / capability 概念
2. `types/chat.types.ts`：定义统一聊天请求/响应模型
3. `interfaces/chat-capability.adapter.ts`：约束 provider 适配器接口
4. `adapters/minimax-chat.adapter.ts`：MiniMax 实现
5. `ai-provider-registry.service.ts`：provider 注册中心
6. `ai-chat.service.ts`：业务侧统一聊天入口
- 价值：
1. provider 参数差异被收敛在 adapter 内部。
2. 业务层未来接入更多模型时只要新增 adapter，不必推倒重写。

### Step 3：新增 Copilot AI 编排服务
- 新增文件：`src/modules/copilot/copilot-agent.service.ts`
- 实现内容：
1. 使用 LangChain `ChatPromptTemplate` 组织输入。
2. 给模型发送“知识引用 + 客户事实 + 会话摘要 + 历史问题”。
3. 要求模型严格输出 JSON。
4. 通过 `zod` 校验结构化结果。
- 价值：
1. 决策结果可验证，不是散乱自由文本。
2. 为后续接工具调用、RAG、记忆提取打下基础。

### Step 4：升级 Copilot 主链路
- 变更文件：`src/modules/copilot/copilot.service.ts`
- 升级点：
1. 先做知识检索与记忆读取。
2. 若 AI 开启，则走 `CopilotAgentService`。
3. 若 AI 失败，则自动退回规则版，保证主链路不中断。
4. 对 AI 输出增加守卫逻辑，例如：
   - 没有 citation 时不允许高置信度直接 reply
   - `clarify` 必须有澄清问题
   - `escalate` 必须有工单建议
5. 将 AI 提取出的稳定客户事实写入长期记忆。
- 价值：
1. 兼顾智能性和可控性。
2. 避免模型返回脏结构或高风险回复直接透出。

### Step 5：更新文档与启动手册
- 更新内容：
1. 本地启动与联调手册增加 MiniMax 开关说明
2. README 增加 AI 适配层说明
3. 新增 LangChain 模型适配文档
- 价值：
1. 新同学不需要读代码也知道怎么跑。
2. 面试讲项目时能解释为什么不是直接把 SDK 写进业务服务。

## 6. 关键设计取舍
1. **为什么不是直接在 `CopilotService` 里 new `ChatMinimax`**：那样业务层会和具体厂商强耦合，后续接其他模型会很痛。
2. **为什么 AI 失败还要回退规则版**：客服场景优先保证“可用”，不能因为外部模型波动直接把主链路打挂。
3. **为什么要求模型只返回 JSON**：后端需要结构化决策结果来做守卫、审计和后续持久化。
4. **为什么加守卫逻辑**：LLM 可以生成文本，但“是否能直接回复客户”仍必须由业务规则兜底。
