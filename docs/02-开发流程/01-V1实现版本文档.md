# 客服 Copilot 实现版本文档

## 1. 文档信息
- 当前版本：`v0.2.0-db`
- 日期：`2026-03-11`
- 维护模块：`apps/agent-service`
- 目的：记录从模板到“数据库化 MVP”的实现路径，便于回看与面试讲解。

## 2. 版本演进

### v0.1.0-mvp（已完成）
1. 完成客服 Copilot 业务闭环。
2. 采用内存仓储（DataStore）快速验证业务。

### v0.2.0-db（本次完成）
1. DataStore 从内存改为 PostgreSQL + TypeORM Repository。
2. 新增数据库实体、迁移历史、CLI 命令。
3. 环境配置从“脚本替换文件”改为 `.env + 校验`。
4. 为 pgvector 预留字段（migration 已创建 vector 列）。

## 3. 本次实现目标
1. 落地可追踪的数据库迁移机制（参考 42service 的 migration 规范）。
2. 消除硬编码配置，统一环境变量入口。
3. 保持业务接口不变，平滑替换存储层。

## 4. 关键实现步骤

### Step 1：重构环境配置体系
- 动作：移除 `scripts/build-env.js` 与 `environment*.ts` 文件替换机制。
- 新实现：
1. `src/config/configuration.ts`：集中读取环境变量。
2. `src/config/env.validation.ts`：启动时校验配置合法性。
3. `ConfigModule.forRoot` 按 `NODE_ENV` 加载 `.env.<env>`。
- 价值：
1. 避免构建时覆写文件带来的风险。
2. 配置可观测、可验证、可迁移。

### Step 2：引入数据库基础设施
- 新增目录：`src/database`
- 新增内容：
1. `database.module.ts`（TypeORM 全局连接模块）
2. `data-source.ts`（TypeORM CLI/migration 配置）
3. `entities/*.entity.ts`（表结构映射）
4. `migrations/*.ts`（建表与扩展历史）

### Step 3：落地 migration 历史
- 新增迁移：`1762950000000-createCustomerCopilotCoreTables.ts`
- 迁移内容：
1. 创建 `vector` 扩展。
2. 创建知识、会话、记忆、工单、审计表。
3. 建立关键索引。
- 价值：
1. 所有数据库变更可追踪、可回滚。

### Step 4：改造 DataStore 为 Repository 实现
- 变更文件：`src/modules/data-store/data-store.service.ts`
- 变更点：
1. 从数组/Map 存储改为 TypeORM Repository 读写。
2. 所有方法改为异步 Promise。
3. 检索策略仍保留 token-overlap 打分（先保证行为一致）。

### Step 5：业务模块同步异步化
- 涉及模块：`knowledge/memory/copilot/tickets/audit`
- 改造点：
1. service/controller 全链路改为 async/await。
2. seed 数据改为 `onModuleInit` 触发。
3. Copilot 链路保持原动作策略不变。

### Step 6：脚本与命令体系升级
- `package.json` 新增：
1. `db:create`
2. `db:gen`
3. `db:run`
4. `db:revert`
5. `db:run:beta|prod`
6. `db:revert:beta|prod`
- 价值：
1. 日常建表/加字段有标准流程。

## 6. 关键设计取舍
1. **为什么仍保留 token 检索**：先完成存储层切换，保障行为稳定，再迭代向量检索。
2. **为什么选 TypeORM**：贴近 Nest 生态，且迁移体系满足“历史可追踪”。
3. **为什么选 PostgreSQL**：与 pgvector 组合自然，便于后续语义检索落地。
